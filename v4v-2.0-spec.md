# V4V 2.0: A Specification for Web Boostagrams

## Abstract

V4V 2.0 is a specification for attaching structured donation metadata (sender, amount, message, source) to Lightning payments made on the web. It builds on the boostagram concept introduced by Podcasting 2.0 but adapts it for the realities of LNURL-pay, the diversity of Lightning wallets, and the Nostr ecosystem.

The core design uses a Nostr kind 30078 event as a public metadata sidecar, joined to the Lightning payment by the bolt11 invoice's `payment_hash`. This sidesteps the wallet-compatibility problems of keysend, the size limits of LNURL comment fields, and the infrastructure requirements of NIP-57 zaps, while preserving anonymous donations, working with any standard LNURL-pay endpoint, and requiring no custom backend on either side.

This document specifies the protocol and explains the design choices behind it.

## Background

A *boostagram* is a Lightning payment with rich attached metadata: who sent it, how much, why, and where they were when they sent it. Podcasting 2.0 popularized the term: listeners boost a podcast episode and the metadata reaches the podcaster's tooling along with the sats.

The Podcasting 2.0 boostagram protocol is built on Lightning *keysend* (BLIP-10). Keysend is a spontaneous-payment primitive: the sender's node sends an HTLC directly to the recipient's node pubkey, with no invoice, and the metadata is attached as TLV records (specifically TLV record 7629169) inside the HTLC payload. The recipient's node decodes the TLVs and the boost arrives complete.

That model works beautifully for the podcast use case, where listeners use podcast-aware apps (Fountain, Castamatic, Podverse, Breez) that all support keysend. But translating it to the broader web for tipping any creator, donating to any project, with any Lightning wallet, this strategy runs into constraints the podcast world only solved by adopting purpose-built sender wallets.

## The Problem

For a static-site web app like a creator's homepage, a project landing page or a donation portal, three constraints define the design space:

**1. Sender wallet diversity.** Donors arrive with whatever Lightning wallet they have. The dominant wallets among the most active donor demographic include Phoenix, Wallet of Satoshi, Cash App, Strike, and most LNURL-only wallets, which do not support keysend. Adopting the Podcasting 2.0 model would mean turning away most donors. This is the deal-breaker for keysend in this context.

**2. Recipient infrastructure variability.** Some recipients run their own Lightning nodes (Start9, Umbrel, raw LND on a server). Some use custodial services (Alby, Wallet of Satoshi). Some use LNbits or Phoenixd or a hybrid. A protocol that requires the recipient to run a specific node type, host a custom invoice-issuing endpoint, or expose their LND to receive TLVs is a non-starter for most recipients.

**3. Static site constraint.** Many recipients don't have a backend at all. A specification that requires server-side invoice generation to embed metadata in a bolt11's `description_hash` ruins this. The protocol needs to work with client-only code talking to a third-party LNURL-pay endpoint.

The intersection of these constraints rules out the obvious approaches:

- **Keysend (Podcasting 2.0)** fails on (#1).
- **Custom backend with `description_hash` linking** fails on (#3).
- **NIP-57 Zaps** fails on (#2): the recipient must run an NIP-57-aware LNURL endpoint. NIP-57 also doesn't support meaningful anonymous donation, since the zap request is signed by the donor's real key.

## The Insight

What's universal across every Lightning wallet that can pay an LNURL invoice? Two things:

1. **The bolt11 invoice's `payment_hash`.** It's a 32-byte cryptographic identifier of the payment. Every node that handles the HTLC sees it, and the recipient's node always records it on settlement, regardless of what intermediate forwarding occurred.

2. **Public Nostr relays.** Every web app that has a Nostr-native identity layer already has Nostr connectivity.

If we use the `payment_hash` as a join key between the Lightning side and the Nostr side, we can store the boost metadata as a public Nostr event keyed by the payment hash. The Lightning payment itself is unchanged, it goes through whatever LNURL flow the donor's wallet supports. The metadata lives on Nostr, free of Lightning's protocol limitations. A recipient can then build a bot to read the `payment_hash` off LND on settlement, query Nostr for the matching event, and get the structured boost data.

This is the V4V 2.0 design.

## Specification

### Event: Kind 30078 Donation Boostagram

V4V 2.0 uses NIP-78 (Application-specific data) kind 30078 as the metadata carrier. Kind 30078 is parameterized-replaceable - addressable by `(kind, pubkey, d_tag)` - and is specifically designed to allow apps to store structured data without claiming a dedicated kind number.

**Required tags:**

- `["d", "<payment_hash_hex>"]` — the 64-character lowercase hex of the bolt11's payment_hash. This is the linkage to the Lightning payment. Implementations MUST NOT publish a kind 30078 unless they have successfully extracted the real payment_hash from the issued bolt11; if extraction fails, the boost MUST be aborted rather than published with a placeholder, since a non-payment-hash `d` tag produces an orphan event no recipient bot can ever match.
- `["app", "<app_name>", "<app_version>"]` — identifies the publishing client.
- `["type", "donation_boostagram"]` — distinguishes V4V 2.0 events from other kind-30078 uses.
- `["sender", "<npub or empty>"]` — the donor's npub, or the empty string for anonymous donations.
- `["recipient", "<lud16>"]` — the Lightning address of the donation recipient.
- `["amount", "<msats>"]` — the amount in millisatoshis as a stringified integer.
- `["url", "<site_url>"]` — the site root from which the boost was made. Must NOT include path or query — `https://example.com` is correct, `https://example.com/some/page?id=42` is wrong. This avoids leaking page-context information into the boost record.

**Content:** free-form text. The donor's optional written message. Empty string when no message was provided.

![Figure 2 - V4V 2.0 Kind 30078 Note Spec Proposal](https://i.nostr.build/uIfIIUziGhOIH93R.png)

**Signing:**

- *Anonymous donations:* signed by a single-use burner keypair (see "Anonymous Donations" below).
- *Attributed donations:* signed by either the donor's real Nostr key (cryptographic attribution) or by a burner key with the donor's npub in the `sender` tag (claimed attribution). Both are valid; clients may choose based on UX trade-offs.

**Publishing:** the event is published to a fixed relay set (see "Relay Set" below) so bots have a known query target.

### Relay Set

V4V 2.0 events are published to four relays:

```
wss://relay.damus.io
wss://nos.lol
wss://relay.primal.net
wss://purplepag.es
```

These relays are chosen for durability, query reliability, and the absence of paywalls or arbitrary filtering. Recipient bots subscribe to this set when looking for boost metadata.

The fixed set is intentional. A floating "publish to the donor's relays" model would force the bot to discover the donor's NIP-65 relay list before each query adding latency and a failure mode for donors with no kind 10002. A small fixed set keeps the bot's query path simple and predictable.

### Anonymous Donations: The Burner Key

V4V 2.0 supports anonymous donation as a first-class mode, not a degraded fallback.

When the donor opts for anonymous mode, the kind 30078 event is signed by a fresh single-use Nostr keypair generated in the donor's browser at boost time. The secret key is held in memory only long enough to produce the event signature, then zeroed and discarded. The `sender` tag is left empty.

This achieves two properties simultaneously:

1. **The signature itself reveals nothing.** The event's `pubkey` field is a pseudorandom value with no on-chain history, no follow graph, no profile metadata. It's not connectable to the donor by any Nostr-native correlation.

2. **Logged-out donations work.** The donor doesn't need a Nostr identity at all. Anyone with a Lightning wallet can donate, and the metadata event still publishes — just with an entirely throwaway pubkey on the signature side.

The burner-key trick is specifically what NIP-57 zaps cannot do: zaps require a signed kind 9734 zap-request from the donor's real key, which renders the donor's identity non-deniable. V4V 2.0 makes anonymity the donor's choice without sacrificing the metadata channel.

### Attribution Without Sacrificing the Channel

When the donor wants attribution, two modes are spec-compliant:

1. **Sign the kind 30078 with the donor's real Nostr key (cryptographic attribution).** The event is verifiably theirs. UX cost: a signer prompt during boost generation. Privacy cost: the event is publicly tied to the donor's pubkey on Nostr.

2. **Sign with a burner key but include the donor's npub in the `sender` tag (claimed attribution).** The donor says it's from them, but the signature doesn't prove it. Avoids the signer-prompt round-trip; relies on the absence of any clear impersonation payoff.

The reference implementation (mynostr) currently exposes a binary anonymous-or-not toggle: anonymous off ⇒ cryptographic attribution (real-key signed, real-npub `sender`); anonymous on ⇒ burner-key signed with empty `sender`. Claimed attribution is reserved as a spec-allowed third mode but is not yet wired into the UI. Future client implementations may expose both attribution modes side-by-side.

### Optional: Kind 1 Share to Feed

Donors who want their boost visible on their public Nostr feed may additionally publish a kind 1 note signed by their real Nostr key. This is independent of the kind 30078 which is explicitly for the recipient's bot; the kind 1 is for the donor's followers.

Recommended kind 1 shape:

**Content:**
```
Just boosted ⚡ <amount> sats to nostr:<recipient_npub>

<donor's optional message>

<site_url>
```

**Tags:**
- `["t", "boost"]`
- `["t", "<app_name>"]` (e.g. `["t", "mynostr"]`)
- `["r", "<site_url>"]`
- `["p", "<recipient_pubkey_hex>"]` — for client mention rendering and recipient notifications.
- `["client", "<app_name>"]`

**Timing:** the kind 1 should be published only after the bolt11's payment is confirmed (via LUD-21 verify polling, or LND-side observation). Publishing it before confirmation risks polluting the donor's feed with announcements for payments that never settled.

**Publish set:** the kind 1 is a regular note and is published to the donor's normal Nostr write relays — *not* the V4V 2.0 fixed relay set. The fixed set is for bot-targeted metadata only. Strict NIP-65 outbox-aware publishing (read the donor's kind 10002, fan out to that exact relay list) is the recommended mode but not required; the reference implementation currently uses its Nostr client's default publish path, which broadcasts to the connected pool. Either is acceptable so long as the kind 1 reaches the donor's followers.

## The Recipient Bot

This section describes how the reference implementation - the bot that publishes mynostr.app's donation receipts - consumes the V4V 2.0 metadata stream and joins it with the recipient's Lightning payment record. The implementation is a small, single-purpose Python script that runs on a self-hosted LAN box, polls the recipient's Lightning node for new settled incoming payments, looks up matching kind 30078 events on the V4V 2.0 relay set, and publishes a kind 1 receipt note signed by the project's Nostr identity.

![Figure 3 - Bot Loop Configuration Overview](https://i.nostr.build/zORQnRMdyL01umAW.png)

### Reference implementation

The reference bot runs on an Ubuntu 24.04 mini PC on the same LAN as the recipient's Lightning node. It is a ~150-line Python 3 script with two direct runtime dependencies in the script itself (`requests` for the node's HTTP API, `websocket-client` for Nostr), plus a shared utility module that handles event signing (which pulls in `pynostr` for nsec parsing + Schnorr signing and `bech32` for npub conversion).

The Lightning node is a [Start9](https://start9.com) server running [Alby Hub](https://github.com/getAlby/hub), which is the current consumer-facing wrapper around an LDK-backed node. The bot does not talk to LND directly; Alby Hub exposes an authenticated HTTP API at `/api/transactions` that returns the same `payment_hash`, `descriptionHash`, and settlement metadata the bot would otherwise pull from `lnrpc.ListInvoices`. The abstraction is convenient: it sidesteps macaroon handling, gRPC tooling, and tor-only onion endpoints for a bearer-token HTTP request from the LAN. A bot targeting raw LND would substitute `lnrpc.SubscribeInvoices` (push) or `ListInvoices` (poll) and read `r_hash` off settled invoices; the rest of the flow is identical.

The bot's identity - the Nostr keypair it signs receipt notes with - lives in a credentials.env file intentionally outside the bot's working directory and outside any version-controlled tree. The script never reads or echoes the file directly; a small `load_config()` helper parses it into a dict at startup. The bot account is a dedicated identity (e.g. `npub1mp9...crk6j` for mynostr.app), distinct from any human's personal Nostr key. 

### The boost-watch daemon

The bot is not a long-running daemon. It is a one-shot script invoked on a fixed cadence by `systemd --user`-style scheduling — a `systemd.timer` unit with `OnUnitActiveSec=10min`, which is functionally equivalent to a cron job at the same interval. Each invocation is fully self-contained: load config, fetch the latest 50 transactions from the node, filter them, query Nostr for matches, publish receipts, persist state, exit. Process lifetime is sub-second in the no-new-payments case and a few seconds when boosts are present.

State between invocations is a single ISO-8601 timestamp in a plain text file (`last_seen.txt`). Each run filters the node's transaction list down to:

- `type == "incoming"`
- `state == "settled"`
- `descriptionHash` is present (i.e. the BOLT11 commits to a hash, not a plain description — this is what the LNURL endpoint produces when issuing a V4V 2.0 invoice)
- `appId == <project_app_id>` (the integer ID Alby Hub assigns to the wallet/connection that received the payment — used to discriminate this project's incoming flow from other apps the same node serves)
- `settledAt > last_seen`

Survivors of the filter are sorted by `settledAt` ascending so multi-boost runs publish in chronological order. After successfully publishing a receipt for a boost, `last_seen` advances to that payment's `settledAt`. Failed kind 30078 lookups are the exception, handled in "Nostr query strategy" below.

The 10-minute cadence is a deliberate trade-off. Faster polling (every minute, or websocket subscriptions) would make receipts feel more real-time but would also produce noisier log output and contend with relay rate limits when no work is needed. Slower polling (hourly) lets too many boosts accumulate between runs, which dilutes the per-event "thank-you" feeling. Ten minutes is the empirical sweet spot for this implementation — also a knowing nod to Bitcoin's average block time, though there is no functional relationship between the two; the choice was driven by the polling-noise vs receipt-latency trade-offs above. Bots with a stronger real-time requirement should switch to a push model on the node side.

### Nostr query strategy

For each filtered candidate, the bot opens a websocket to each relay in the V4V 2.0 fixed publish set — `wss://relay.damus.io`, `wss://nos.lol`, `wss://relay.primal.net`, `wss://purplepag.es` — and sends a `REQ` with the filter:
```json
{ "kinds": [30078], "#d": ["<payment_hash_hex>"] }
```
Relays are queried serially, not in parallel. The first relay to return an `EVENT` short-circuits the loop: the bot closes the websocket and uses that event. If a relay sends `EOSE` with no matching event, the bot moves to the next relay. Per-relay socket timeout is 10 seconds, which covers the realistic worst case for a healthy relay; any relay that exceeds this is logged and skipped. If no relay in the set returns an event, the candidate is dropped from this run.

Querying the full publish set rather than a subset costs at most one round-trip per missing relay, but it removes the failure mode where the bot's narrow query set diverges from where the publisher actually wrote. The query relay list is imported from a shared module (`V4V_RELAYS` in `shared/boost_formatter.py`) so it stays in lockstep with the spec — a relay-set change is one edit in one place.

The current implementation retries failed lookups implicitly across runs: when a candidate's kind 30078 isn't found on any relay in the publish set, the bot does not advance `last_seen` past that payment, so the same `payment_hash` is re-queried on the next 10-minute fire. The 10-minute cadence makes propagation races rare in practice (the spec's 30-second propagation envelope is well below the polling interval), but the bot is forgiving in the cases that do occur.

One edge case: if a *later* boost in the same run successfully publishes, the watermark advances to that later `settledAt`, and any earlier failed-lookup boost is no longer eligible for retry. This is a deliberate trade-off — it prevents the bot from getting permanently stuck on payments that will never produce a kind 30078 (e.g. direct tips to the lightning address rather than V4V 2.0 boosts).

### Action triggers

On a successful match, the bot does exactly one thing: publish a single kind 1 note from the project's Nostr account that announces the boost.

The note is composed deterministically from the kind 30078 event:

```
⚡ New Donation on <project name>
💰 <amount in sats, comma-formatted>
👤 nostr:<sender_npub>     (or "👤 Anon" if the sender tag is empty)
💬 "<message, with @npub/@nevent/@naddr mentions rewritten to nostr:... URIs>"
🔗 <url, suppressed if it contains 'localhost'>

#<project_tag> #Bitcoin #V4V #V4V2
```

Mention rewriting normalizes any `@npub1...`, `npub1...`, `@nevent1...`, `nevent1...`, `@naddr1...`, or `naddr1...` in the donor's free-form message to the canonical `nostr:` URI form so Nostr clients render them as proper inline mentions. The regex used has lookbehinds to skip already-prefixed entities and to skip matches glued to a preceding word character.

Beyond the formatted content, the receipt note carries NIP-57 zap split tags constructed from the boostagram's `sender` and the publishing account: equal-weight `["zap", <hex>, <relay>, "1"]` entries for each party that has a `lud16` in their kind 0 profile. With both parties present and reachable that is a 50/50 split, so any zap landing on the receipt note pays the original donor as much as the project. Anonymous boosts (empty `sender` tag) or donors without a public lud16 collapse the split to 100% to the publisher. This is a reference implementation choice, not a V4V 2.0 mandate; future implementations may pick other split structures or omit the splits entirely.

The note is built as an unsigned event tuple `[0, pubkey, created_at, kind, tags, content]`, NIP-01-serialized, hashed to produce the event `id`, and signed with the project's nsec using a Schnorr signature over the secp256k1 curve. The signed event is then broadcast to the publishing account's NIP-65 outbox: the bot reads the account's latest kind 10002 and publishes to the relays it advertises (those without a `read` marker). This is the project's own outbox, distinct from the V4V 2.0 fixed query set above — the receipt is for followers of the project's account, who will look for it on the publisher's relays. Accounts without a published kind 10002 fall back to a small hardcoded relay set for kind 1 specifically; replaceable events refuse to fall back at all, since they need to land at the author's outbox to be discoverable. Per-relay publish failures are logged but don't block the rest of the broadcast.

### Self-hosting notes

The reference setup runs on a Start9 server (Alby Hub) and a separate Ubuntu mini PC (the bot host) on the same LAN. The split is intentional: the Lightning node and its hot key sit on one purpose-built device, and the bot, which holds the project's Nostr nsec but no Lightning keys, sits on a general-purpose machine. The bot reaches the node via its `.local` LAN address and the Start9 root CA cert pinned in the bot host's trust store.

For Umbrel, RaspiBlitz, or any setup with a node-attached HTTP API, substitute the API base URL and bearer token. For raw LND without Alby Hub, swap the `requests.get(/api/transactions)` call for an `lnrpc.SubscribeInvoices` stream or `ListInvoices` paginated call against the LND gRPC interface, the rest of the script is unchanged.

## Privacy Considerations

V4V 2.0 publishes metadata to public relays. The privacy model is:

**What is exposed:**

- For anonymous boosts: amount, message, recipient, source URL, payment_hash. None of these are joinable to the donor's identity through Nostr-native correlation.
- For attributed boosts (claimed or cryptographic): all of the above, plus the donor's npub.

**What is NOT exposed:**

- Lightning routing details (who relayed the HTLC, which channels were used).
- The donor's wallet provider, unless the bolt11 itself reveals it.
- The donor's IP address, beyond what Nostr relays already see for any of their activity.

**Correlation surfaces:**

- Anyone observing Lightning gossip and querying public Nostr relays can join `payment_hash → metadata`. For attributed boosts this means they can confirm "this payment was associated with this npub."
- The `payment_hash` itself is not secret in Lightning — anyone routing the HTLC sees it. V4V 2.0 does not increase Lightning's existing exposure of `payment_hash`; it just attaches metadata to it.

**For high-privacy donations:** anonymous mode plus a Lightning wallet that doesn't bind the donor to a public identity (e.g., a fresh Phoenix wallet, or coins routed through a privacy-preserving channel) provides reasonable cover. The kind 30078 event signed by a burner key has no link to the donor through Nostr-native means.

## Why Not Just Put the Nostr Event Reference in the Lightning Comment?

A simpler-looking variant would be: include an `nevent1...` reference in the LNURL comment field (so the bolt11's description carries `[mynostr boost] — nostr:nevent1xxx`), publish a kind 1 (or kind 30078) on Nostr with the actual metadata, and have the bot fetch it by the nevent. The Lightning comment becomes the bridge instead of `payment_hash`.

This would work, but only when the recipient controls the LNURL endpoint. If the recipient's lud16 is on a third-party service (Alby, Wallet of Satoshi, hosted LNbits), the comment lives in that service's database; the recipient's LND only sees the forwarded HTLC with no description.

The `payment_hash` approach works regardless of LNURL hosting topology: the recipient's LND always sees the payment hash on settlement. This is the architectural reason V4V 2.0 chose the Nostr-keyed-by-payment-hash model over the LNURL-comment-carries-Nostr-reference model.

If the recipient *does* control their LNURL endpoint, the simpler variant becomes available, and arguably cleaner. Future implementations may offer both modes: comment-carries-nevent for self-hosted recipients, payment-hash-keyed for everyone else. V4V 2.0 picks the universally-applicable path as the default.

## Future Work

### Backend-issued invoices with bidirectional `description_hash` linking

The most spec-clean evolution would be a recipient-controlled invoice-issuing backend that bakes a Nostr event reference into the bolt11's `description_hash`. Specifically: the backend computes `description_hash = sha256(canonical_form_of(kind_30078_event))` and includes that in the issued invoice. The bolt11 then carries a verifiable cryptographic reference to the metadata event.

This eliminates the asymmetric "donor publishes first, bot looks up later" model in favor of a fully cryptographic bidirectional link. It requires the recipient to control the LNURL endpoint, which not all recipients do. V4V 2.0 leaves this as an opt-in evolution path; the static-site default remains the `payment_hash`-keyed sidecar.

### bolt12 offers

bolt12 is the long-term protocol successor to bolt11 and LNURL-pay. It includes structured metadata fields natively and removes many of the limitations LUD-12 comments impose. As wallet support for bolt12 matures, a future V4V version may target bolt12 directly and obsolete the Nostr sidecar entirely. This is years away.

### Spec consolidation

The "V4V 2.0" naming acknowledges the conceptual descent from Podcasting 2.0's boostagram model while signaling the architectural break (LNURL+Nostr versus keysend+TLV). If the protocol gains adoption beyond this implementation, formalizing it as a Nostr Improvement Proposal (NIP) with a dedicated kind range would replace the kind-30078-with-namespacing-tags approach.

## Open Questions and Limitations

- **No bidirectional cryptographic link.** The bolt11 has no awareness of the kind 30078 event. A bad actor could forge a kind 30078 with a correct `payment_hash` and false metadata, hoping a bot displays it. Bots should validate that the `recipient` tag matches their own lud16 before trusting any other field.

- **`payment_hash` on a public relay.** As noted in the privacy section, this creates a small public correlation surface. Acceptable trade-off for the discovery benefits.

- **Custodial-wallet recipients get no benefit.** Recipients who use Strike, Wallet of Satoshi, or other custodial services do not have access to their underlying Lightning records, including the `payment_hash`. For them, V4V 2.0 metadata exists on Nostr but is operationally inaccessible. They get the same UX as today, "received N sats" with no context. V4V 2.0 doesn't worsen their position; it just doesn't help.

- **LNURL endpoint variability.** Some LNURL-pay servers may not advertise the fields we expect, may impose unusual comment limits, or may modify the description in unexpected ways. V4V 2.0 implementations should degrade gracefully when LNURL metadata is unusual.

- **Race condition on instant payments.** The donor publishes the kind 30078 before paying, but instant Lightning payments may settle before the metadata propagates across the four-relay set. Bot implementations should retry the Nostr query a few times before treating the absence of metadata as definitive. A short retry window (e.g. 2-5 seconds between attempts, up to 30 seconds total) handles realistic propagation delays.
